# 沐龙量化交易平台（ApexAI Quant）— 系统架构与功能介绍

> 面向技术面试的项目说明文档。涵盖整体架构、功能模块、并发模型、技术栈与典型数据流。

---

## 一、项目定位（30 秒电梯演讲）

这是一个 **个人/小团队自用的全栈量化交易平台**，覆盖：

- **加密 + 美股 + A 股** 多市场
- **信号接收 → AI 评分 → 实盘执行 → 监控告警 → 研究展示** 完整链路
- **Web 管理端** + **TradingView Webhook** + **钉钉推送** 三端协同

核心设计思想：**交易路径与 AI 分析路径解耦**（旁路架构），Webhook 低延迟执行不阻塞 DeepSeek 评分；**多进程部署**避免单点阻塞。

### 2.2 三进程部署（生产环境）

脚本：`tools/start_trading_stack.sh`

| 进程 | 端口 | 职责 | 并发模型 |
|------|------|------|----------|
| `run_api.py` | 8100 | 主平台 API、前端静态资源、内置调度器 | **asyncio** (Uvicorn) |
| `webhook_service.py` | 8005 | TradingView 低延迟交易路由 | **asyncio** + 每实例 Lock |
| `tradingview_bot.py` | 5001 | 信号格式化、多群钉钉、评分旁路 | **Flask 同步** + daemon Thread |

**为什么要拆进程？**

1. Webhook 交易要求 **毫秒级响应**，不能与重 API（回测、AI 分析）抢 GIL
2. `tradingview_bot` 独立运行，TradingView 只打 5001，不影响 8100 重启
3. DeepSeek 评分、快讯轮询等 **CPU/IO 密集任务** 与交易隔离

## 三、并发与线程模型（面试高频）

本平台 **混合使用** 多种并发模型，按场景选型：

### 3.1 asyncio（FastAPI 主事件循环）

- **Uvicorn** 驱动 FastAPI，处理 HTTP 请求
- **startup 定时任务**（`api/main.py`）：
  - 每 4 小时 Hyperliquid K 线同步
  - 北京时间 **05:00** 研究卡片现价刷新
  - 每 1 小时加密上涨 Top50 扫描缓存
  - 每周六 **10:00** 美股泡沫 DeepSeek 分析
  - 快讯 / Polymarket 监控从磁盘 **restore** 后台线程
- **阻塞 IO 隔离**：`asyncio.to_thread()` 把同步函数（拉 Yahoo、写 JSON、LightGBM 推理）丢到线程池，避免卡死 event loop

### 3.2 threading.Thread（daemon 后台轮询）

适用于 **长周期轮询 + 同步 HTTP** 的监控类服务：

| 服务 | 文件 | 行为 |
|------|------|------|
| 自选快讯 | `core/stock_news_alert.py` | 每 30s 轮询金十/雅虎等，命中关键词钉钉推送 |
| Polymarket 概率 | `core/polymarket_alert.py` | 轮询 Gamma/CLOB API |
| 币种分钟预警 | `core/binance_monitor.py` | Binance K 线 + 订单簿异动 |
| 链上活跃度 | `core/chain_activity_monitor.py` | ETH/Arb/BSC RPC 15 分钟窗口对比 |
| MACD 形态 | `core/macd_pattern_monitor.py` | 多 symbol × 多周期 MACD 形态检测 |
| AI 评分旁路 | `core/crypto_signal_scorer.py` | Webhook 触发后 fire-and-forget 评分 |

**单例模式**：`get_*_instance()` / `set_*_instance()` 保证全局唯一后台线程，API 启停通过 `/start` `/stop` 控制。

### 3.3 Thread + 独立 asyncio event loop（策略实例）

`api/routers/trading.py` 中策略实例管理：

```python
HYPE_STRATEGY_INSTANCES: Dict[str, HYPEAdaptiveShortStrategy] = {}
HYPE_STRATEGY_THREADS: Dict[str, threading.Thread] = {}
```

每个策略：**daemon Thread → 内部 `asyncio.new_event_loop()` → 策略主循环**。  
策略之间互不阻塞，可并行跑 HYPE 做空 + ETH 趋势 + 自适应做多。

### 3.4 subprocess（Backpack/Deepcoin 实盘）

`engine/live_trading.py` 对部分交易所用 **子进程** 启动独立 Python 实盘脚本，与主 API 进程隔离，崩溃不影响 Web 服务。

### 3.5 asyncio + Lock（Webhook 交易引擎）

`webhook_service.py`：

- 多实例 `engine_instances: Dict[str, WebhookTradingEngine]`
- 每实例 `asyncio.Lock` 防止并发信号重复下单
- `asyncio.create_task(engine.execute_signal())` 非阻塞执行

### 3.6 跨进程文件锁（DeepSeek 去重/限流）

`crypto_signal_scorer.py` 使用 **文件锁** `deepseek_score_usage.lock`：

- 多进程（bot + run_api + webhook）共享日调用配额
- 120 秒内同 symbol+action+周期 **去重**，避免重复计费

## 四、功能模块详解

### 4.1 策略交易 & Webhook 实盘

**页面**：`/trading`  
**API**：`/api/trading`  
**核心文件**：

- `api/routers/trading.py` — 策略 CRUD、启动/停止、Webhook 端点
- `webhook_service.py` — 独立 Webhook 微服务
- `engine/webhook_trading.py` — Ostium 引擎
- `engine/hyperliquid_trading.py` — Hyperliquid 引擎
- `engine/live_trading.py` — Backpack/Deepcoin 子进程实盘

**支持交易所**：Backpack、Deepcoin、Ostium（Arbitrum）、Hyperliquid、Binance、Lighter

**策略类型**：

| 策略 | 说明 |
|------|------|
| HYPE 自适应做空 | 4H 周期，Hyperliquid |
| ETH 趋势做空 | 2H 周期 |
| 自适应做多/做空 | TradingView Webhook 驱动 |
| 自动平仓 | 条件触发平仓 |
| 网格交易 | 多档位挂单（`/grid-trading`） |

**典型数据流（TradingView → 实盘）**：

```
TV Alert JSON
  → webhook_service :8005（HMAC 签名校验）
  → classify 实例（按 strategy_name / symbol 路由）
  → WebhookTradingEngine.execute_signal() [asyncio]
  → Ostium/HL REST 或 Web3 下单
  → MySQL orders / positions / trades 落库
  → 可选：同步 AI 评分 → 钉钉
```

**adaptive-long 特殊路径**：买入信号 **先同步完成 AI 评分**，再决定是否开仓（评分与开单门槛可配置）。

## 六、认证与安全

- **JWT Bearer**：登录后 `localStorage.token`，Axios 拦截器自动带 Header
- **Webhook HMAC-SHA256**：`webhook_service` 签名校验防伪造信号
- **密钥隔离**：私钥/API Key 在 `.env` 或实例配置，**不入 MySQL**
- **功能开关**：`ENABLE_OKX_TRADE`、`CRYPTO_SCORE_ENABLED` 等环境变量
- **钉钉关键词**：消息正文须含「提醒」以通过机器人安全策略

## 七、典型面试问答准备

### Q1：为什么不用单体 FastAPI，而要三进程？

> Webhook 交易对延迟敏感，不能与 AI 评分、快讯轮询、回测抢 GIL。拆进程后 8100 重启不影响 8005 接单，5001 钉钉 bot 独立运维。DeepSeek 用文件锁跨进程去重，避免 bot 和 API 重复调用。

### Q2：AI 评分如何不影响下单？

> **旁路架构**：评分在 daemon Thread 中 fire-and-forget；adaptive-long 买入路径可选「同步评分再开仓」，但评分失败/超时策略可配置是否拦截。加密/美股通过 `signal_asset_router` 分流到不同 scorer。

### Q3：监控服务如何保证重启后恢复？

> 配置写 JSON + MySQL singleton；`api/main.py` startup 调用 `try_restore_from_disk()`；用户主动 stop 会设 `user_stopped` 标志，防止误恢复。

### Q4：如何处理 DeepSeek 费用失控？

> ① `deepseek-v4-flash` + thinking disabled；② 跨进程文件锁日限额 + 120s 去重；③ 本机默认 `CRYPTO_SCORE_ENABLED=0`；④ `usage.json` 记录 host/pid/billed_model 便于审计；⑤ 三进程一键重启脚本确保新代码生效。

### Q5：前端如何与后端交互？

> 开发：Vite `:8050` proxy `/api` → `:8100`。生产：FastAPI 挂载 `frontend/dist`，SPA fallback。React Router 客户端路由 + JWT 401 跳登录。

### Q6：你负责的核心模块是哪些？（按实际填写）

> 示例：加密/美股 AI 评分流水线、自选快讯监控池、DeepSeek 费用治理、TradingView 三进程部署、2026 策略矩阵年化展示逻辑等。

## 十、部署与环境变量

| 变量 | 作用 |
|------|------|
| `TRADING_SERVER=1` | 服务器模式 |
| `CRYPTO_SCORE_ENABLED=0/1` | 加密 AI 评分开关 |
| `US_STOCK_SCORE_ENABLED=0/1` | 美股 AI 评分开关 |
| `DEEPSEEK_SCORE_MODEL` | 默认 deepseek-v4-flash |
| `DEEPSEEK_SCORE_THINKING=0` | 关闭 thinking 降费用 |
| `ENABLE_OKX_TRADE` | OKX 控制台是否允许交易 |
| `HTTP_PROXY` / `HTTPS_PROXY` | 国内服务器访问 Yahoo/DeepSeek |
| `STOCK_NEWS_DINGTALK_WEBHOOK` | 快讯钉钉 Webhook |
